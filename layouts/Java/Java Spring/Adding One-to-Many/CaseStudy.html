    Case Study: Driver's License

    In this lesson, we will be walking through a hypothetical application that will allow users to create licenses for every person. The important questions will arise as you build the one-to-many assignments in the next section. However, this one-to-many case will be an introduction to how relationships in general are set up and used in Spring.

    Learning Objectives:

    • Get familiar with relationships through an example of a 1:1 relationship
    • Review how object instances can be member variables in a different model
    • Understand how member variables map relationships in the database
    • Implement a relationship using input from a form and data binding

    Consider the following ERD and wireframe:

    Showing License Info for a Person

    In the last lesson we provided code for setting up a 1:1 relationship between two models. Let's take a closer look at the new code that was introduced:

    models/Person.java 

    @OneToOne(mappedBy="person", cascade=CascadeType.ALL, fetch=FetchType.LAZY)
private License license;

    models/License.java

    @OneToOne(fetch=FetchType.LAZY)
@JoinColumn(name="person_id")
private Person person;

    So, let's say in our controller, we fetch a person out of the database, calling to our service and to our repository, etc. What would that look like? And how might we get that person's name and license info, onto the page?

    Assuming you have your service set up with basic CRUD methods, it might look something like this in your controller:

    @GetMapping("/persons/{person_id}")
public String showPerson(@PathVariable Long person_id, Model model) {
    
    Person someAwesomePerson = personService.findById(person_id);
    model.addAttribute("person", someAwesomePerson);
    
    return "showPerson.jsp";
}

    But what about the license info? Do we also need to call the License service and add it to the view model to render on the page? The answer is no! Because the Person model has a license member variable, we can fetch it through the person using dot notation like so:

    <!--- ... --->
<h1>Person Details:></h1>
<table class="table">
    <thead class="thead-dark">
        <tr>
            <th>Name</th>
            <th>License Number</th>
            <th>State</th>
            <th>Exp Date</th>
        </tr>
    </thead>
    <tbody>
        <tr>
            <td>
                <c:out value="${person.firstName}"/>
                <c:out value="${person.lastName}"/>
            </td>
            <td><c:out value="${person.license.number}"/></td>
            <td><c:out value="${person.license.state}"/></td>
            <td>
                <c:out value="${person.license.expirationDate}"/>
            </td>
        </tr>
    </tbody>
</table>
<!--- ... --->

    Data Binding with One-to-Any!

    No, that's not a typo, as this can apply to 1:1 and 1:n.

    Now, how would you set up a 1:1 relationship between a particular person and a particular license in the first place? Well, we're going to use data binding for creating a new license, just as we have for prior assignments. As you can see from the wireframe, you don't need a license to create a person, but you do need a person to create a license. We have a drop-down menu that we will populate with all the persons (okay people) in order to select the one whose license we are creating. But what values are we sending to the server through that drop-down select input? 

    <!--- inside the form:form --->
    <!--- Drop down select menu --->
    <form:select path="person">
        <c:forEach var="onePerson" items="${persons}">
            <!--- Each option VALUE is the id of the person --->
            <form:option value="${onePerson.id}" path="person">
            <!--- This is what shows to the user as the option --->
                <c:out value="${onePerson.firstName}"/>
                <c:out value="${onePerson.lastName}"/>
            </form:option>
        </c:forEach>
    </form:select>
<!--- ... --->

    As long as you have the empty object model attribute in your form (data binding) you can either send back the id as the value or the object itself. Both will bind that particular person to the license and set the foreign key to the person_id. And the POST will look like this:

    @PostMapping("/licenses")
public String licenses(@Valid @ModelAttribute("license") License license) {
    // error handling with binding result omitted    
    licenseService.create(license); // Already has the person!
        
    return "redirect:/persons";
}
