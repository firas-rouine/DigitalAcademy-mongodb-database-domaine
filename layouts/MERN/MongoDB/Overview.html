    Database and MongoDB

    Objectives

    • Define what the database is in a web application
    • Begin introduction to MongoDB

    Defining the Data

    A web application is composed of three parts.

    The client code that appears inside a browser. This is the HTML, CSS, and JavaScript that make up the appearance and functionality of an application.

    The server code that responds to the actions of the client. This can be done in many languages, as long as they can process HTTP requests coming from browsers.

    Finally, we have data persistence. An email client only works because the usernames, passwords, messages, and timestamps are all saved to a database. Once saved, the data can persist across multiple sessions. This is our database layer. A database is a collection of stored and organized data that we can access through our application. Databases are often classified by how that data is modeled.

    The most common operations done with a database is creating, manipulating, saving, and accessing data. Different databases prioritize different operations, some favoring fast reading and writing, where others prioritize scaling at an enterprise level, with hundreds of thousands of connected users.

    Documents and MongoDB

    JavaScript objects can be used to document data.

    // User
{
    '_id': '507f191e810c19729de860ea'
    'first_name': 'Lia',
    'last_name': 'Swift',
    'age': 44,
    'favorite_colors': ['red', 'blue', 'pink'],
    'address': {
        'street': 'fake street',
        'building': 123,
        'city': 'Anytown',
        'state': 'CA'
    }
}

    This format is sometimes called json or JavaScript Object Notation. We can also call this object a document. We can think of a document as one unit of data. Documents are defined by their own structure. If we consider the document above, we can see that much of the data is readable text.

    MongoDB specializes in documents. It simplifies the way we access documents compared to many databases. By default, MongoDB will attach an Object _id to each of our documents, so they can be found by that _id quickly. Doing this makes MongoDB fast for both reading and writing to the database at very high volumes. The downside is we lose accuracy in how we can request data from the database. It might be very quick and easy to get all the users from our User collection, or one user by it's _id, but getting just users with a first_name of 'Lia' would be very expensive. Searching and filtering data are not MongoDB's strongest features. However, we can save a lot of time by doing those operations on the client side with a little bit of JavaScript instead.
